## 3. 본인이 직접 정리한 결과물 평가

### 3-1. 회원가입 & 결과 페이지

#### 필수 요구사항

- [x] 이름·아이디·비밀번호·이메일·약관 동의에 `required` 적용
- [x] 아이디에 `pattern`을 적용하여 영문과 숫자 4~12자만 허용
- [x] 비밀번호에 `minlength="8"` 적용
- [x] 전화번호에 `010-0000-0000` 형식의 `pattern` 적용

#### 세부 요구사항

- [x] 성별은 `radio`로 구성하여 하나만 선택
- [x] 관심 분야는 `checkbox`로 구성하여 복수 선택
- [x] 지역은 `select`로 구성
- [x] 상단 헤더를 `position: fixed`와 `z-index`로 고정
- [x] 제출 시 `result.html`로 이동
- [x] 결과 페이지에서 입력값을 `table`로 표시
- [x] 관심 분야 복수 값을 결과에 모두 표시

#### 동작 테스트

- [x] 빈 필수값 제출 시 브라우저 경고와 제출 차단 확인
- [x] 아이디 2자는 패턴 위반, 4자 이상은 통과하는지 확인
- [x] 비밀번호 3자 입력 시 8자 이상 안내 확인
- [x] 관심 분야 “프론트엔드”와 “디자인”이 결과에 함께 표시되는지 확인
- [x] 정상 제출 시 `result.html`의 표에 각 값이 정확히 표시되는지 확인
- [x] 결과 페이지를 스크롤해도 상단 헤더가 고정되는지 확인

#### 자기평가

- 잘된 점:
  1. 입력값 검증을 JavaScript에만 의존하지 않고 `required`, `pattern`, `minlength`, `type`, `autocomplete` 등 HTML이 제공하는 속성을 활용하여 웹 표준에 맞는 회원가입 폼을 구현한 점이 잘되었다고 생각합니다.
  2. `form`, `label`, `input`, `select`, `table` 등의 태그를 목적에 맞게 사용하면서 폼의 제출 과정과 각 요소의 역할을 이해하고, 화면 모양뿐 아니라 문서의 의미가 드러나도록 구조를 작성한 점이 잘되었다고 생각합니다.
  3. 사용자의 입장에서 입력 과정에서 불편할 수 있는 부분을 고민하고, 필수 입력 표시, 입력 조건 안내, 형식 예시, 자동완성, 버튼 hover와 모바일 반응형 디자인을 추가하여 사용 편의성을 높인 점이 잘되었다고 생각합니다.
  4. 개인정보가 URL과 방문 기록에 노출될 가능성을 줄이기 위해 `sessionStorage`로 데이터 전달 방식을 변경하고, 비밀번호 원문을 저장하지 않도록 처리하면서 데이터 전달 과정의 보안 문제를 고민하고 개선한 점이 잘되었다고 생각합니다.
  5. 성별, 관심 분야, 지역의 표시 문구를 상수로 분리하여 반복되는 문자열을 줄이고 수정하기 쉽게 구성한 점이 잘되었다고 생각합니다.
- 어려웠던 점과 해결 과정:
  1. 처음에는 `form`의 `action`으로 결과 페이지에 이동하면서 입력값을 URL 쿼리스트링으로 전달했는데, 이름·이메일·비밀번호와 같은 개인정보가 주소창과 방문 기록에 그대로 노출되는 문제가 있었습니다. 단순 실습이더라도 개인정보 노출 가능성을 줄이고 싶었으며, 제출 이벤트에서 입력값을 `sessionStorage`에 저장한 뒤 결과 페이지에서 꺼내 표시하는 방식으로 변경하는 과정이 어려웠습니다.
  2. `sessionStorage`도 JavaScript로 접근할 수 있어 비밀번호 원문을 저장하는 것은 적절하지 않다고 판단하였습니다. 비밀번호 대신 입력 여부만 `hasPassword`로 저장하고, 결과 화면에는 사용자가 입력한 실제 비밀번호 길이와 관계없이 고정된 별표 8개를 표시하도록 구현하였습니다. 데이터 전달 기능뿐 아니라 어떤 정보를 저장하고 화면에 노출할지 결정하는 부분이 어려웠지만, 민감정보를 다룰 때 고려해야 할 점을 경험할 수 있었습니다.
- 개선할 점:
  1. 기능 구현에 집중해 전체적인 디자인이 단순합니다. 색상과 여백, 입력 요소의 상태별 디자인을 보완하고 싶습니다.
  2. 성별, 관심 분야, 지역의 선택 항목이 HTML에 반복되어 있어 수정할 때 각각 변경해야 합니다. 선택 항목을 JavaScript에서 생성하여 한 곳에서 관리하고 싶습니다.
  3. 현재는 약관 내용을 확인하지 않고 동의할 수 있습니다. 약관의 마지막까지 이동해야 동의할 수 있도록 하고, 약관 확인 과정의 편의성을 높이기 위해 `맨 아래로 이동` 버튼도 제공하고 싶습니다.
  4. 현재는 비밀번호를 한 번만 입력하므로 오타가 있어도 바로 확인하기 어렵습니다. 비밀번호 확인 입력란을 추가하고 두 값의 일치 여부를 검증할 수 있도록 개선하고 싶습니다.
- 실습 이후 추가한 과제/기능:
  1. 사용자 편의성을 고려해 MDN 문서를 참고하여 회원가입 입력 필드에 `autocomplete` 속성을 적용
  2. 일관된 크기 계산과 기본 여백 제거를 위해 전역 CSS 초기 스타일을 적용
  3. 사용자가 필수 입력 항목을 직관적으로 구분할 수 있도록 라벨에 빨간 별표와 필수 입력 안내 문구를 적용
  4. 운영체제별 글꼴 차이를 줄이고 일관된 한글 UI를 제공하기 위해 Google Fonts의 Noto Sans KR을 적용
  5. 개인정보가 URL 쿼리스트링에 노출되는 것을 방지하기 위해 `sessionStorage`로 입력값을 전달하고, 민감정보인 비밀번호는 원문 대신 입력 여부만 저장하여 결과 화면에 고정된 별표(`********`)로 표시하도록 개선
  6. 저장된 회원가입 정보 없이 결과 페이지에 직접 접근하면 안내 메시지를 표시하고 회원가입 페이지로 이동하도록 예외 처리
  7. 사용자가 입력 조건과 형식을 미리 확인할 수 있도록 각 입력란에 안내 문구와 형식 예시를 `placeholder`로 제공

#### 결과 확인 테스트

- [x] 이미지 로딩을 차단하거나 개발자 도구로 모든 `img`에 의미 있는 `alt`가 있는지 확인

#### 자기평가

- 잘된 점:
  1. `header`, `nav`, `main`, `section`, `article`, `aside`, `footer`를 각 영역의 역할에 맞게 사용하여 페이지의 구조와 의미가 드러나도록 작성한 점이 잘되었다고 생각합니다.

  3. 고정 내비게이션의 링크를 각 섹션의 `id`와 연결하고 부드러운 스크롤과 맨 위로 이동 기능을 구현하여 긴 페이지에서도 원하는 내용을 편리하게 찾을 수 있도록 한 점이 잘되었다고 생각합니다.

#### 자기평가

- 잘된 점:
  1. 할 일을 `{ id, text, done }` 객체 배열로 관리하고 추가·완료·삭제·필터 기능이 상태 변경 후 목록을 다시 렌더링하는 동일한 흐름을 따르도록 구현했습니다. `push`, `find`, `filter` 등의 배열 메서드를 실제 상태 관리에 적용하면서 배열 상태와 화면 렌더링의 관계를 이해한 점이 잘되었다고 생각합니다.
  2. 화면과 이벤트 처리는 `app.js`, 데이터 저장과 복원은 `storage.js`, 비동기 문구 요청은 `apis.js`로 역할을 구분했습니다. `import/export`로 모듈을 연결하고 `localStorage`에 JSON 형식으로 데이터를 저장하여 새로고침 후에도 상태가 유지되도록 구현한 점이 잘되었다고 생각합니다.
  3. 오늘의 한마디 기능을 처음에는 `Promise` 객체와 `setTimeout`을 사용한 Mock API 형태로 구현하여 비동기 응답을 직접 만들어 보았습니다. 이후 정적 웹 서버도 파일 요청에 HTTP로 응답하므로, 데이터를 로컬 `quotes.json` 파일로 분리하면 `fetch`로 불러올 수 있다는 점을 알게 되었습니다. 이에 직접 만든 Mock API를 제거하고 `fetch`, 응답 상태 확인, JSON 변환, `async/await`, `try/catch`를 사용하는 구조로 개선했습니다. 한 가지 방식에 머무르지 않고 두 가지 비동기 데이터 처리 방식을 직접 시도하고 비교한 점이 잘되었다고 생각합니다.
  4. 사용자 입력을 `innerHTML`에 직접 넣으면 HTML 태그나 스크립트성 속성이 실행되는 DOM 기반 XSS 취약점이 발생할 수 있다는 점을 확인했습니다. 할 일 요소는 `createElement()`로 생성하고 사용자 입력은 `textContent`로 설정하여 HTML이 아닌 일반 문자열로 처리되도록 개선함으로써, 사용자 입력을 통한 XSS 위험을 줄인 점이 잘되었다고 생각합니다.
  5. `localStorage`의 데이터가 잘못된 JSON 형식이면 `JSON.parse()` 오류로 앱 실행이 중단될 수 있다는 점을 고려했습니다. 데이터 복원 과정을 `try/catch`로 처리하고, 복원에 실패하면 오류를 콘솔에 남긴 뒤 빈 배열을 반환하여 나머지 기능은 계속 사용할 수 있도록 안정성을 높였습니다.
  6. 클래스를 활용해 할 일 불러오기, 저장, 추가, 삭제, 완료 상태 변경 기능을 `TodoManager`라는 하나의 객체로 묶었습니다. 할 일 데이터와 관련된 역할을 한곳에서 관리하도록 구성하여 화면 처리 코드와 데이터 관리 코드의 책임을 명확히 나누고, 기능을 수정하거나 추가할 때 관련 코드를 쉽게 찾을 수 있어 유지보수하기 좋게 만든 점이 잘되었다고 생각합니다.
  7. `TodoManager` 클래스를 적용하고 나니 여러 인스턴스가 생성될 경우 각 인스턴스가 서로 다른 할 일 상태를 가져 데이터가 일관되지 않거나 동작이 꼬일 수도 있겠다고 생각했습니다. 현재 과제에서는 실제로 발생할 가능성이 작지만, 이를 예방하는 방법을 학습하기 위해 싱글톤 패턴을 적용했습니다. 생성자에서 기존 인스턴스가 있는지 확인하고 생성된 인스턴스만 모듈에서 내보내 앱 전체가 하나의 할 일 상태를 공유하도록 구현한 점이 잘되었다고 생각합니다.
- 어려웠던 점과 해결 과정:
  1. 완료 토글 기능을 구현하면서 처음에는 각 체크박스에 이벤트 리스너를 직접 등록했습니다. 하지만 처음 조회한 요소에만 이벤트가 등록되어 이후 동적으로 추가되거나 다시 렌더링된 할 일의 체크박스에서는 상태 변경 코드가 실행되지 않는 문제가 있었습니다. 새 항목을 생성할 때마다 이벤트를 다시 등록할 수도 있지만 코드가 반복되고 관리하기 어려워질 수 있었습니다. 자식 요소에서 발생한 클릭 이벤트가 이벤트 버블링을 통해 부모 요소까지 전달된다는 점을 학습한 뒤, 각 체크박스 대신 부모인 할 일 목록에 이벤트 리스너를 한 번만 등록했습니다. 이후 `event.target`과 `closest()`로 실제 클릭된 체크박스와 해당 할 일 항목을 찾고 상태를 변경하도록 구현하여 기존 항목과 새로 추가된 항목을 같은 이벤트 리스너로 처리할 수 있었습니다.
  2. 처음에는 `todos.length`를 새 할 일의 ID로 사용했지만, 중간 항목을 삭제한 뒤 새로운 항목을 추가하면 기존 할 일과 ID가 중복되는 문제가 있었습니다. `nanoid`와 같은 라이브러리도 고려했지만 작은 과제에 별도 의존성을 추가하는 것은 불필요하다고 판단했습니다. 대신 브라우저에서 기본으로 제공하는 `crypto.randomUUID()`를 사용하여 삭제나 배열 순서와 관계없이 각 할 일에 고유한 문자열 ID를 부여했습니다. `dataset`에서 읽은 값도 문자열이므로 별도의 숫자 변환 없이 객체의 ID와 바로 비교하도록 수정했습니다.
  3. 과제에서 `fetch`와 `async/await` 사용을 요구했지만 호출할 실제 API가 없어 처음에는 `Promise` 객체 안에서 `setTimeout`으로 응답 지연을 만든 Mock API를 구현했습니다. 이 방식으로 비동기 처리 흐름은 연습할 수 있었지만 실제 HTTP 요청이 발생하지 않아 `fetch`를 활용할 수 없다는 한계가 있었습니다. 해결 방법을 찾는 과정에서 Live Server와 같은 정적 웹 서버도 클라이언트의 파일 요청에 HTTP로 응답하는 서버라는 점을 알게 되었습니다. 명언 데이터를 `quotes.json`으로 분리한 뒤 해당 경로를 `fetch`로 요청하도록 변경했고, 직접 만든 Mock API는 제거했습니다. 그 결과 별도의 백엔드 API 없이도 실제 요청과 응답 상태 확인, JSON 변환, 예외 처리까지 구현할 수 있었습니다. 처음에는 `navigator.onLine`으로 오프라인 상태를 검사하는 방법도 적용했지만, 이 값은 브라우저의 네트워크 연결 여부만 나타낼 뿐 특정 요청의 성공을 보장하지 않고 인터넷이 끊겨도 로컬 서버의 JSON은 요청할 수 있다는 점을 확인했습니다. 따라서 이 검사를 제거하고, 실제 `fetch` 실패와 HTTP 오류, JSON 변환 오류가 발생했을 때 `try/catch`에서 기본 문구를 표시하도록 수정했습니다.
- 개선할 점:
  1. 현재는 문구 데이터 요청 과정에서 발생한 오류를 모두 같은 기본 문구로 처리하고 있습니다. 추후에는 요청 실패, HTTP 오류, JSON 형식 오류를 구분하여 원인에 맞는 안내를 제공하고 싶습니다.
  2. 현재는 할 일의 내용과 완료 여부만 관리하여 언제까지 처리해야 하는지 확인하기 어렵습니다. 할 일 객체에 마감 날짜를 추가하고, 날짜가 임박했거나 지난 항목을 구분하여 표시하는 기능을 구현하고 싶습니다.
  3. 현재 목록은 할 일을 추가한 순서대로만 표시되어 사용자가 중요도에 따라 순서를 바꿀 수 없습니다. 드래그 앤 드롭으로 항목의 순서를 변경하고, 변경된 순서도 `localStorage`에 저장되도록 개선하고 싶습니다.
  4. 할 일이 많아지면 서로 다른 목적의 항목이 한 목록에 섞여 원하는 항목을 찾기 어려울 수 있습니다. 할 일 객체에 카테고리를 추가하고 업무·학습·개인 일정처럼 목적별로 구분하거나 필터링할 수 있도록 개선하고 싶습니다.
- 실습 이후 추가한 과제/기능:
  1. 전체·진행 중·완료 필터 결과가 비어 있을 때 현재 필터 상태에 맞는 안내 문구를 표시하여 사용자가 빈 화면의 의미를 알 수 있도록 개선
  2. 사용자 입력을 통한 DOM 기반 XSS 위험을 줄이기 위해 할 일 요소를 `createElement()`로 생성하고 입력 내용은 `textContent`로 안전하게 표시
  3. 손상된 `localStorage` 데이터로 인해 앱 실행이 중단되지 않도록 데이터 복원 과정에 `try/catch`와 빈 배열 폴백을 적용
